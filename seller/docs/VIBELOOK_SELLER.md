# 1. Введение и цели продукта

**Vibelook Seller** — веб-кабинет для селлеров одежды (РФ, WB/Ozon), который:

1. **Автоматически подтягивает товары** из WB/Ozon + даёт ручную загрузку.

2. **Классифицирует вещи по стилям**:
   - Streetwear
   - Minimalism
   - Classic Modern
   - Boho
   - Athleisure
   - Romantic
   - Rock/Soft Grunge

3. **Генерирует реальные луки** только из существующих товаров (своих и других селлеров).

4. **Создаёт контент под луки**:
   - изображения + простые видео (слайд-шоу) через n8n;
   - автопостинг из наших аккаунтов Vibelook во все соцсети;
   - платную рекламу через наши рекламные кабинеты.

5. **Показывает селлеру**:
   - индекс узнаваемости бренда (WB + Яндекс Wordstat);
   - базовую рекламную и продуктовую аналитику;
   - агрегированную аналитику по профилю аудитории (рост, размеры, цветотипы).

# 2. Технологический стек

## Frontend
- **Next.js / React** — единый проект с текущим Vibelook
- **TypeScript** — типизация
- **CSS Modules / Tailwind** — стилизация (наследуется от основного проекта)

## Backend

### Next.js API Routes / Server Actions
- API и бизнес-логика
- Обработка запросов от фронтенда
- Валидация данных
- Управление сессиями

### Python-сервисы (отдельные)
- **Работа с WB/Ozon API**: синхронизация товаров, обновление данных
- **Рекламные API**: интеграции с Facebook, VK, Яндекс, TikTok
- **Индексы и аналитика**: Яндекс Wordstat, WB поиск, агрегация метрик

## База данных
- **Supabase (PostgreSQL)** — основная база данных
- **Supabase Storage** — хранение файлов (изображения, видео)

## Оркестрация/автоматизация
- **n8n** (отдельный инстанс) для:
  - AI-классификации товаров по стилям
  - Генерации контента (изображения, видео)
  - Автопостинга в соцсети
  - Синхронизации данных
  - Модерации контента

## Авторизация
- **Telegram Login / OAuth** — через Telegram-бота
- Создание/поиск записи seller по `tg_user_id`
- Управление сессиями через Supabase Auth

## Инфраструктура
- Развёртывание: Vercel (Next.js) / отдельный сервер (Python-сервисы)
- n8n: отдельный инстанс (self-hosted или cloud)

# 3. Роли и доступ

## 3.1. Роли

### Seller (единственная роль на MVP)
- Создаётся при первом логине через Telegram
- Имеет доступ к своему кабинету и данным
- Управляет своими товарами, луками, кампаниями

## 3.2. Авторизация

### На лендинге кабинета
- Кнопка **«Войти через Telegram»**
- Интеграция с Telegram Login Widget

### После успешного Telegram Login
1. Создаётся/находится запись `seller` в БД
2. Создаётся/обновляется связка `tg_user_id`
3. Проверяется статус тарифа:
   - Если trial активен → редирект на `/dashboard`
   - Если trial закончился → редирект на `/billing/lock`
   - Если оплачен → редирект на `/dashboard`

### Статусы пользователя
- `trial_active` — активный пробный период
- `trial_ended → locked` — пробный период закончился, доступ заблокирован
- `paid → active` — активная подписка

# 4. Тарифы и биллинг

## 4.1. Тарифы

### 1. Start (~2000 ₽/мес)

**Включено:**
- 1 ключ WB + 1 ключ Ozon
- Один профиль селлера
- Ленд Vibelook: размещение его луков
- Индекс узнаваемости (WB + Wordstat)
- Базовая аналитика по лукам (просмотры/клики/переходы)

**Не включено:**
- Без автопостинга
- Без платного трафика
- Без пополнения баланса

### 2. Pro

**Включено:**
- Несколько ключей WB/Ozon (до 3–5)
- Всё из Start
- Автопостинг во все соцсети
- Доступ к рекламным кампаниям «Продажи»/«Тест товара»
- Возможность пополнить баланс (внутренний кошелёк)
- Покупка пакетов генераций луков (после бесплатных 5)

### 3. Enterprise (от 100 000 ₽/мес)

**Включено:**
- Расширенные лимиты (кол-во товаров, луков, кампаний, генераций контента)
- Большие бюджеты на трафик
- Индивидуальные настройки (флаг `is_enterprise`)

## 4.2. Платежи

### Подписка
- Оформление тарифа → привязка карты (через провайдера)
- Раз в месяц автоматическое списание
- Управление подпиской через Stripe/ЮKassa/аналог

### 14-дневный пробный период
- Стартует с момента первой активации кабинета
- По окончании:
  - Кабинет полностью блокируется — показывается только экран выбора тарифа и оплаты
  - Нет доступа к аналитике/лукам до оплаты

### Баланс (кошелёк, только Pro+)
- Селлер пополняет баланс на сумму в рублях
- Из баланса списывается:
  - перепродажа трафика (рекламный spend),
  - покупка пакетов генераций луков

# 6. Сущности и модель данных

Имена полей в стиле `snake_case` (для Postgres).

## 6.1. Таблица `sellers`

```sql
CREATE TABLE sellers (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
    updated_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
    tg_user_id TEXT UNIQUE NOT NULL,
    brand_name TEXT,
    contact_email TEXT,
    tariff_plan TEXT NOT NULL DEFAULT 'trial', -- 'trial' | 'start' | 'pro' | 'enterprise'
    trial_ends_at TIMESTAMP WITH TIME ZONE,
    is_blocked BOOLEAN DEFAULT FALSE, -- когда триал кончился и не оплачен
    wb_api_keys JSONB DEFAULT '[]', -- массив объектов {name, api_key}
    ozon_api_keys JSONB DEFAULT '[]',
    balance_rub NUMERIC(10, 2) DEFAULT 0, -- баланс кошелька (Pro+)
    free_look_generations_left INTEGER DEFAULT 5,
    is_enterprise BOOLEAN DEFAULT FALSE
);
```

**Описание полей:**
- `id` — уникальный идентификатор
- `tg_user_id` — ID пользователя Telegram (unique)
- `brand_name` — название бренда
- `tariff_plan` — текущий тариф
- `trial_ends_at` — дата окончания пробного периода
- `is_blocked` — флаг блокировки (если trial закончился)
- `wb_api_keys` / `ozon_api_keys` — массив API-ключей
- `balance_rub` — баланс кошелька (для Pro+)
- `free_look_generations_left` — количество бесплатных генераций

## 6.2. Таблица `products`

Один продукт = один артикул (можно с несколькими размерами).

```sql
CREATE TABLE products (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    seller_id UUID NOT NULL REFERENCES sellers(id) ON DELETE CASCADE,
    created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
    updated_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
    source TEXT NOT NULL, -- 'wb' | 'ozon' | 'manual'
    source_sku TEXT, -- SKU из источника
    nm_id TEXT, -- для WB
    product_id TEXT, -- для Ozon
    title TEXT NOT NULL,
    description TEXT,
    category TEXT, -- 'top' | 'bottom' | 'shoes' | 'accessory' | 'outerwear' | 'dress' | 'other'
    gender TEXT, -- 'male' | 'female' | 'unisex'
    age_group TEXT, -- 'adult' | 'teen' | 'kids'
    color TEXT,
    price_original NUMERIC(10, 2),
    price_with_discount NUMERIC(10, 2),
    size_chart JSONB, -- полные замеры (ОГ/ОТ/ОБ, длина и т.д.)
    size_range TEXT, -- короткая форма, например "XS-XL"
    image_urls JSONB DEFAULT '[]', -- массив строк URL
    style_tags JSONB DEFAULT '[]', -- массив стилей: streetwear, minimalism, classic_modern, boho, athleisure, romantic_soft, rock_soft_grunge
    is_active BOOLEAN DEFAULT TRUE,
    allow_cross_looks BOOLEAN DEFAULT FALSE -- разрешить использовать в чужих луках
);
```

**Описание полей:**
- `source` — источник товара (WB/Ozon/ручной)
- `style_tags` — массив тегов стилей
- `size_chart` — детальная информация о размерах
- `allow_cross_looks` — разрешение на использование в луках других селлеров

## 6.3. Таблица `looks`

```sql
CREATE TABLE looks (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    seller_id UUID NOT NULL REFERENCES sellers(id) ON DELETE CASCADE,
    created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
    product_ids JSONB NOT NULL DEFAULT '[]', -- массив product_id; могут быть товары разных селлеров
    main_image_url TEXT, -- итоговая картинка лука (Supabase Storage)
    video_url TEXT, -- слайд-видео (Supabase Storage, nullable)
    style_tags JSONB DEFAULT '[]', -- массив стилей лука
    usage_contexts JSONB DEFAULT '[]', -- массив: work, home, sport, evening, casual, beach, street
    is_unisex_variant_male BOOLEAN DEFAULT FALSE,
    is_unisex_variant_female BOOLEAN DEFAULT FALSE,
    is_published BOOLEAN DEFAULT FALSE, -- опубликован на лендинге Vibelook
    moderation_status TEXT DEFAULT 'pending', -- 'pending' | 'approved' | 'rejected' | 'review_needed'
    moderation_reason TEXT -- причина отклонения/предупреждения
);
```

**Описание полей:**
- `product_ids` — массив ID товаров в луке
- `main_image_url` / `video_url` — ссылки на медиафайлы
- `usage_contexts` — контексты использования лука
- `moderation_status` — статус модерации

## 6.4. Таблица `campaigns` (реклама)

```sql
CREATE TABLE campaigns (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    seller_id UUID NOT NULL REFERENCES sellers(id) ON DELETE CASCADE,
    created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
    updated_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
    name TEXT NOT NULL,
    look_ids JSONB NOT NULL, -- минимум 3 для тестов
    goal TEXT NOT NULL, -- 'sales' | 'test_ctr'
    budget_rub NUMERIC(10, 2) NOT NULL, -- min 5000
    status TEXT NOT NULL DEFAULT 'draft', -- 'draft' | 'running' | 'paused' | 'finished' | 'error'
    started_at TIMESTAMP WITH TIME ZONE,
    ended_at TIMESTAMP WITH TIME ZONE,
    ads_platforms JSONB DEFAULT '[]', -- список площадок
    tracking_params JSONB, -- utm параметры
    external_campaign_ids JSONB DEFAULT '{}' -- ID кампаний в рекламных системах
);
```

**Описание полей:**
- `goal` — цель кампании (продажи или тест CTR)
- `budget_rub` — бюджет (минимум 5000 ₽)
- `ads_platforms` — массив платформ (Facebook, VK, Яндекс, TikTok)
- `external_campaign_ids` — ID кампаний во внешних системах

# 7. Интеграции

## 7.1. WB / Ozon API

### Python-сервис

**Функции:**
- По сохранённым API-ключам селлера:
  - подтягивает список товаров, их атрибуты, размеры, цены, остатки
- По расписанию обновляет данные (например, раз в сутки)
- Обрабатывает ошибки и логирует проблемы

### В кабинете

**Экран "Интеграции" (`/integrations`):**
- Поля для добавления/проверки ключей WB и Ozon
- Кнопка "Проверить ключ" → результат (OK / ошибка)
- Кнопка "Синхронизировать сейчас" → запуск импорта
- Показ статуса:
  - последняя синхронизация
  - импортировано товаров
  - ошибки импорта

### Процесс синхронизации

1. Селлер вводит API-ключ → сохраняется в `sellers.wb_api_keys` / `ozon_api_keys`
2. Python-сервис получает список товаров через API
3. Для каждого товара:
   - Проверка существования по `source_sku` / `nm_id`
   - Создание или обновление записи в `products`
   - Загрузка изображений в Supabase Storage
4. После завершения → отправка Telegram-уведомления

## 7.2. Яндекс Wordstat + WB поиск

### Python-скрипты / n8n-флоу

**Функции:**
- По `brand_name` собирают частоты запросов:
  - в WB (по их поиску/статистике, если есть возможность)
  - в Yandex Wordstat
- Раз в N дней обновляют `brand_awareness_index`

### Генерация рекомендаций

**Примеры рекомендаций:**
- «Ваш бренд ищут <100 раз в месяц — уровень локального магазина. Рекомендуем запустить тестовую кампанию.»
- «Рост запросов +30% за последние 30 дней. Имеет смысл усилить луки и постинг.»

## 7.3. Рекламные платформы

### Общие аккаунты Vibelook

Все кампании крутятся с общих аккаунтов Vibelook (не селлера).

### Python-слой + n8n

**Создание кампаний:**
- Цели: `sales` / `test_ctr`
- Бюджет (от 5000 ₽)
- Креативы из луков (картинки/видео)
- UTM метки для трекинга

**Сбор метрик:**
- Регулярно забирает метрики:
  - impressions (показы)
  - clicks (клики)
  - spent (потрачено)
  - conversions/clicks to WB/Ozon/лендинг

**Платформы:**
- Facebook Ads
- VK Ads
- Яндекс.Директ
- TikTok Ads
- (при необходимости — другие)

# 8. Основные пользовательские сценарии (MVP)

## 8.1. Onboarding селлера

1. Селлер переходит на страницу кабинета → жмёт **«Войти через Telegram»**
2. После логина:
   - Создаётся запись `seller` (статус `trial`, `trial_ends_at = now() + 14 дней`)
   - Проверяется существование по `tg_user_id`
3. Первый экран (`/dashboard`):
   - Предложение подключить WB/Ozon (API ключи)
   - Краткое описание шагов: «Импорт товаров → Генерация луков → Контент/реклама»

## 8.2. Импорт товаров

1. Селлер вводит WB API key / Ozon API key на странице `/integrations`
2. Жмёт **«Импортировать товары»**
3. Python-сервис тянет карточки, записывает в `products`
4. Дополнительно селлер может на отдельной вкладке (`/products/add`) загрузить ручной товар:
   - Заполняет форму (обязательные поля)
   - Загружает фото (в Supabase Storage)

## 8.3. Авто-классификация по стилям

**После загрузки/обновления товара:**

1. n8n запускает AI-процесс:
   - По основному фото определяет стиль(и):
     - streetwear / minimalism / classic_modern / boho / athleisure / romantic_soft / rock_soft_grunge
2. Записывает в `products.style_tags`

## 8.4. Генерация лука

1. Селлер открывает раздел **"Луки"** (`/looks`)
2. Нажимает **"Сгенерировать лук"** (`/looks/generate`)
3. Интерфейс:
   - Выбор набора (например, худи + низ + обувь + аксессуар)
   - Опционально выбор стиля (streetwear и т.д.)
4. Система:
   - Подбирает комбинацию из:
     - его товаров + товаров других селлеров, разрешивших использование
   - Генерит компоновку:
     - AI-изображение (коллаж с вещами / стилизованное изображение)
     - Опционально слайд-видео (через n8n из этих картинок)
   - AI-модерация:
     - проверка на соответствие политике Facebook (без откровенного, без запрещённого контента)
     - оценка "продающее/непродающее"
5. UI:
   - Показывает сгенерированный лук
   - Селлер может:
     - **Сохранить** (лук идёт в каталог и на лендинг)
     - **Перегенерировать** (потратить ещё одну попытку)
6. Лимиты:
   - первые 5 генераций бесплатные (уменьшается `free_look_generations_left`)
   - если 0 → показать поп-ап **«Купить пакет из 20 луков за N ₽»**

## 8.5. Автопостинг

1. Селлер выбирает сохранённый лук
2. Нажимает **«Создать пост»** (`/posting`)
3. Система:
   - Генерирует текст поста по шаблону (без ручного редактирования)
   - Генерирует набор картинок/видео (либо использует уже созданные)
4. Селлер выбирает соцсети (для Pro+):
   - Instagram, Facebook, VK, Telegram, Pinterest, Яндекс.Дзен, TikTok
5. Нажимает **«Опубликовать сейчас»** или **«Запланировать»**
6. n8n отправляет посты в аккаунты Vibelook, не селлера

## 8.6. Запуск рекламы

1. Селлер выбирает:
   - либо 1 лук (для продаж)
   - либо набор минимум из 3 луков/креативов (для теста CTR)
2. Выбирает режим кампании (`/campaigns/create`):
   - **«Продажи»** (оптимизация под переходы/продажи)
   - **«Тест товара»** (малый бюджет, фокус на CTR)
3. Вводит бюджет (ручной, но не меньше 5000 ₽)
4. Для Pro+:
   - проверяется баланс → если недостаточно → предложить пополнить
5. Кампания создаётся в рекламных аккаунтах Vibelook:
   - цели, аудитории, UTM — по умолчанию из пресетов
6. Статусы и метрики тянутся в `campaigns` + `metrics_aggregate`

# 15. API эндпоинты

## Системные страницы для разработчиков

### Продукты

- `GET /api/products` — список товаров селлера
- `POST /api/products` — создание товара (ручной)
- `PUT /api/products/:id` — обновление товара
- `DELETE /api/products/:id` — удаление товара
- `POST /api/products/sync` — запуск синхронизации WB/Ozon

### Луки

- `GET /api/looks` — список луков
- `POST /api/looks` — сохранение лука
- `POST /api/looks/generate` — генерация нового лука
- `PUT /api/looks/:id` — обновление лука
- `DELETE /api/looks/:id` — удаление лука

### Кампании

- `GET /api/campaigns` — список кампаний
- `POST /api/campaigns/create` — создание кампании
- `PUT /api/campaigns/:id` — обновление кампании (пауза/запуск)
- `DELETE /api/campaigns/:id` — удаление кампании

### Постинг

- `GET /api/posts` — список постов
- `POST /api/posting/send` — отправка поста в соцсети
- `PUT /api/posts/:id` — обновление поста (планирование)

### Интеграции

- `POST /api/wb/fetch` — получение данных из WB API
- `POST /api/ozon/fetch` — получение данных из Ozon API
- `POST /api/integrations/check-key` — проверка API-ключа

### Метрики

- `GET /api/metrics` — получение метрик (агрегированных)
- `POST /api/metrics/pull` — обновление метрик из рекламных систем

### Модерация

- `POST /api/moderation/check` — проверка контента на модерацию

### Аналитика

- `GET /api/analytics/brand-awareness` — индекс узнаваемости
- `GET /api/analytics/audience` — профиль аудитории
- `GET /api/analytics/looks` — аналитика по лукам

### Биллинг

- `GET /api/billing/subscription` — информация о подписке
- `POST /api/billing/subscribe` — оформление подписки
- `POST /api/billing/cancel` — отмена подписки

### Кошелёк

- `GET /api/wallet/balance` — текущий баланс
- `POST /api/wallet/deposit` — пополнение баланса
- `GET /api/wallet/transactions` — история транзакций

### Настройки

- `GET /api/settings` — настройки селлера
- `PUT /api/settings` — обновление настроек
